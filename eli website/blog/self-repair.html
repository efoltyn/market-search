Self-Repair Is an Emergent Feature

2026-01-13 • Eli

“Fixing itself” sounds like magic until you look at the interface. Self-repair doesn’t come from a smarter paragraph. It comes from closing the loop between intent and reality.

Models don’t debug text. They debug feedback.

The loop that creates repair

Eli doesn’t just ask the model for an answer. It asks for a plan, commands to run, and diffs to apply — then it runs them and feeds the outputs back in.

plan → run → fail → observe → patch → run → pass

Why “failure” is the ingredient

When a model is constrained to tools, it can’t hand-wave a result. It either produces data and code that runs, or it gets an error. That error is the training signal at inference time.

Most assistants avoid failure by avoiding execution. The output looks smooth — and stays wrong.

Diffs make fixes precise

Repairs work because the surface area is small: a diff is a concrete edit in a concrete file. That lets the model iterate without “forgetting” what it changed.

Undo also matters. If a patch makes things worse, you don’t argue about it — you revert it.

Finance is a brutal teacher

Markets are adversarial. If you ask “is this bullish?”, you can write a persuasive story either way. But if you ask “what happened to returns after the earnings print?” you need prices, timestamps, and arithmetic.

That’s why Eli’s finance tools exist: pull raw numbers first, then interpret them.

The punchline

Self-repair isn’t a feature you bolt onto an LLM. It’s what you get when the system can execute, observe, and edit with a tight feedback loop.

Once the loop exists, smaller models start behaving like debuggers.
